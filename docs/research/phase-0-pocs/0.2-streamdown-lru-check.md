# POC 0.2 — Streamdown 代码块高亮是否复用 `code-block.tsx` 的 Shiki LRU

**结论先行：未复用（完全独立的两套实现）。** 聊天消息流的 Streamdown 高亮路径走 `@streamdown/code`，其内部自建无上限 `Map` 缓存并各自调用 `shiki.createHighlighter`，与 `src/components/ai-elements/code-block.tsx` 的 `highlighterCache` / `tokensCache` 无任何关联。

---

## [仓库事实]

1. `code-block.tsx` 定义了两层 LRU：
   - `highlighterCache = new LRUMap<..>(10)` — `src/components/ai-elements/code-block.tsx:164-167`
   - `tokensCache = new LRUMap<string, TokenizedCode>(200)` — `src/components/ai-elements/code-block.tsx:170`
   - 唯一消费者：同文件导出的 `highlightCode()`（`:251-315`）和 `CodeBlockContent`（`:502`、`:514`）。

2. 聊天消息渲染挂载的是 Streamdown，不走 `code-block.tsx`：
   - `src/components/ai-elements/message.tsx:19` `import { createCodePlugin } from "@streamdown/code"`
   - `:326` `const _codePlugin = createCodePlugin()`；`:336` `streamdownPlugins = { cjk, code: safeCode, math, mermaid }`
   - `:340-347` `<Streamdown plugins={streamdownPlugins} ... />`
   - 整个 `message.tsx` 无 `code-block.tsx` import、无 `components={...}` 覆盖。

3. `PreviewPanel.tsx` 同样仅用 `@streamdown/code`：
   - `src/components/layout/panels/PreviewPanel.tsx:25` 懒加载 `@streamdown/code`
   - `:32` `code: codeMod.code`（即 `createCodePlugin()` 默认实例）

4. 跨仓库搜索 `from "@/components/ai-elements/code-block"` 无匹配 —— `code-block.tsx` 的导出在运行时路径上是死代码（仅用于旧的非 Streamdown 场景或未接入）。

## [外部事实]

`node_modules/@streamdown/code/dist/index.js`（已格式化提取）：

- `createHighlighter` 直接从 `shiki` 导入（`import {bundledLanguages,createHighlighter} from 'shiki'`）。
- 模块级三个 `Map`：`a`（highlighter 缓存，键 `lang-light-dark`）、`h`（tokens 缓存，键 `lang:light:dark:len:head:tail`）、`l`（回调订阅）。**全部无容量上限、无淘汰策略**。
- `createCodePlugin()` 返回的 `highlight()` 读取 `h`，未命中时走 `b()` → `createHighlighter({themes, langs:[lang]})` 并写入 `a`；tokenize 后写入 `h`。
- streamdown 的 `CodeBlock`（`node_modules/streamdown/dist/code-block-37QAKDTI.js` 末尾）通过 `l.highlight({code, language, themes}, cb)` 调用该插件，即上层渲染 = 插件内置 Map，与 `code-block.tsx` 的 `LRUMap` 完全正交。
- `dist/index.d.ts:53` 签名：`createCodePlugin(options?: CodePluginOptions): CodeHighlighterPlugin`，无接受外部缓存/highlighter 的入参。

## [结论 & 改造建议]

**未复用。** 两套高亮器并存时，单次会话可能同时存在两份 Shiki highlighter 实例 + 两份 tokens 缓存；且 `@streamdown/code` 的 Map 无上限，长会话下存在无界增长风险。

**改造动作（按成本递增）：**

1. **推荐（低成本）**：在 `message.tsx` 封装自定义 `code` 插件取代 `@streamdown/code` 的 `createCodePlugin()`，让其 `highlight()` 直接转发到 `code-block.tsx` 导出的 `highlightCode()`，复用既有 `highlighterCache`(10) + `tokensCache`(200)。`CodeHighlighterPlugin` 接口在 `@streamdown/code/dist/index.d.ts:18-39` 已公开，自实现无需改上游。注意：插件的 `highlight` 返回 `TokensResult`，而 `highlightCode()` 返回 `TokenizedCode`（仅 `tokens/fg/bg`，缺 `rootStyle`/`themeName`），需要 shim 成 `TokensResult` 形状（补 `themeName: lightTheme`、构造 `rootStyle`）。
2. **次优**：保留 `@streamdown/code` 但给其 Map 套一层 LRU 包装（需 fork 或 monkey-patch `a/h`，维护成本高，不推荐）。
3. **顺带清理**：若选方案 1，`code-block.tsx` 的 highlight 部分可保留作为共享底座；其 UI 部分（折叠、复制按钮）继续作为直接调用场景使用，不受影响。

**验收信号**：改造后只应存在一份 `createHighlighter()` 调用路径，DevTools Memory 快照里 Shiki highlighter 实例数在语言 ≤10 时稳定为 1 份/(lang,theme) 组合。
